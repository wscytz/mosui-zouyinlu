# v5 开发准备说明

> v4 已完成内容工厂；v5 的目标不是继续堆数量，而是让内容被玩法、构筑、视觉和发布质量吸收。

## v5 开工前门槛

进入 v5 正式开发前，先确认：

1. `npm run test:all` 全绿。
2. `npm run test:automation` 全绿。
3. `npm run skill:check` 全绿。
4. `npm run audit:content` 为 0 errors；warnings 必须有明确处置。
5. 当前本地 commit 已记录；push 失败只作为网络同步问题，不阻塞本地开发。

## Claude 下一轮优先做什么

推荐按这个顺序给 Claude 派活：

1. **v5 主题定稿**：从“流派成型 + Boss 里程碑 + 发布收口”里选一个主标题，不要同时开太多方向。
2. **构筑审计**：按标签统计 191 遗物的实际可玩路线，找出“数量多但没有核心件”的流派。
3. **Boss/精英框架评估**：先写设计文档和测试切片，再动 `game.js` 核心战斗。
4. **视觉冒烟增强**：把封面、首页、战斗、遗物选择、结算页纳入截图检查。
5. **发布前体验清单**：移动端横屏、按钮可点区域、Wiki/图鉴同步、APK/网页资源同步。

## v5 不建议一开始做的事

- 不要先批量再加 100 个遗物。
- 不要直接重构 `game.js` 主循环。
- 不要让 agent 直接写主文件或直接提交。
- 不要在没有 fixture/测试的情况下扩展 merger 能力。
- 不要让视觉资源只进 `www/` 或 `android/`；源资产必须进 `assets/`。

## v5 内容生产原则

v5 可以继续用 Scheme B，但每批内容必须回答三个问题：

1. 它强化了哪个流派？
2. 它有没有进入 `RELIC_RULES` / `BUILD_PREFS` / 测试 / CSS 图标？
3. 它是否让玩家选择时更清楚，而不是只是多一个名字？

如果答案不清楚，先走 idea/design，不要直接产 block。

## 管线复发问题

这些问题已被纳入 v5 前置规则：

- HTML entity 是输出通道问题，不靠 prompt 解决；block rules 会拦截，必要时跑 `npm run fix:entities`。
- CSS 图标优先用稳定形状和 `clip-path`；方向边框容易触发 position-like 禁令。
- skill 同步必须检查整个 skill 目录，不只检查 `SKILL.md`。

## 建议版本切片

- **v5.0-prep**：测试、文档、自动化、视觉冒烟补齐。
- **v5.1-builds**：流派权重、冷标签核心件、遗物选择体验。
- **v5.2-boss**：Boss/精英战机制和测试。
- **v5.3-release**：网页/APK/封面/截图/README/Wiki 收口。

## 交付定义

v5 的“完成”不是内容数量，而是：

- 玩家能感到至少 4 条稳定流派。
- Boss 或精英战有明确记忆点。
- 新内容不靠手感猜测，能被测试和审计覆盖。
- 网页端和 APK 资源同步可靠。
- 文档能让下一轮 Claude 直接接手。
