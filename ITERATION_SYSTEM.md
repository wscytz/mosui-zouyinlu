# 墨祟：走阴录 可持续迭代机制 v13.1

> 目标：让项目后续可以持续交给 Claude / Codex / agent 接手，而不是每轮靠聊天记忆重新解释。本文规定版本节奏、任务切分、验收门槛、复盘和文档更新。

## 一、迭代总原则

### 1. 一个版本只解决一个主问题

每个小版本必须选择一个主主题：

| 类型 | 适合版本 | 例子 |
|---|---|---|
| 稳定版 | v13.1.x | 修选择逻辑、source、对象池、崩溃 |
| UI 版 | v13.2.x | 卡面、HUD、图标、墨夜表现 |
| 性能版 | v13.3.x | 长跑、对象上限、移动端性能 |
| 平衡版 | v13.4.x | 构筑冷门、数值曲线、Boss 节奏 |
| 模块化 | v14.0 | 软拆分 + 数据集中表 + 审计 P1/P2（✅ 已完成） |
| 文档口径 | v15.0-precheck | 文档标题修正 + 移动端预检 + 测试基线（只验收，不加内容） |
| 内容版 | v15.1 - v15.4 | 墨符令 MVP → 补完 → 通用遗物 → 试炼模式 → APK |
| 文档版 | 任意 `.docs` | 文档索引、规范、路线、交接 |

禁止在同一版本里同时做“修 bug + 大内容 + 大重构 + UI 大改”。

### 2. 先收口，再扩展

每个阶段固定顺序：

1. 收集问题。
2. 排优先级。
3. 修 P0 / P1。
4. 补回归测试。
5. 做小范围体验优化。
6. 更新文档。
7. 复盘并决定下一阶段是否允许加内容。

### 3. 事实源优先级

当文档冲突时：

1. 脚本输出和代码实际状态。
2. `PROJECT_INTRO.md`。
3. `ROADMAP.md`。
4. `STRUCTURE_RULES.md`。
5. `DEVELOPMENT.md` / `ARCHITECTURE.md`。
6. `DEVDOC.md` 历史记录。
7. 历史任务包。

## 二、版本节奏

### v13.1 结构治理

目标：让现有内容稳定、可测、可维护。

允许：

- 修选择逻辑、source、对象池、生命周期。
- 补回归测试。
- 更新文档和协作规范。
- 小型无行为变化重构。

禁止（v13.1.x 稳定版语境，v14+ 已部分解禁）：

- ~~新增第 6 武器。~~ v15.1 解禁（墨符令）
- ~~新增第 5 Boss。~~ v14.0 已加（墨魇）

### v13.2 UI 与图标管线

允许：

禁止：

### v13.3 性能与移动端发布

目标：让后期不卡、移动端可测、APK 有前置门槛。

允许：

- debug HUD。
- 长跑对象峰值记录。
- 粒子、弹体、浮字、火场上限。
- 移动端横屏和右摇杆调优。

禁止：

- 移动端未验证就打 APK。

完成条件：

- 固定种子长跑 3 分钟无错误。
- 无尽 10 波以上对象数可控。
- 移动端横屏可玩。
- APK 前置清单完成。

### v13.4 平衡与构筑审计

目标：让 213 遗物不是散点，而是能形成稳定构筑生态。

允许：

- 审计冷标签、死权重、过强联动。
- 微调遗物权重和数值。
- 补少量构筑缺口。
- 增加结算构筑识别。

禁止：

- 大量新增遗物掩盖构筑问题。
- 没有实测就重砍核心流派。

完成条件：

- 输出构筑审计报告。
- 明确 S/A/B/C 流派。
- 至少修复 3 条冷门但有潜力的构筑线。

## 三、任务流

### 1. 收集

每轮开始先整理：

- 用户痛点。
- 上轮未解决问题。
- 测试失败。
- 运行时错误。
- 文档冲突。
- 体验问题。

输出格式：

```md
## 本轮问题池
- P0：
- P1：
- P2：
- 暂缓：
```

### 2. 切分

每个任务必须小到能独立验收：

- 一个 bug。
- 一个系统入口。
- 一个 UI 区域。
- 一个测试类别。
- 一批 20-30 个图标。

不允许“优化一下游戏”这种任务直接开工。

### 3. 执行

每个任务必须写清：

- 类型。
- 允许改动。
- 禁止改动。
- 实现规格。
- 验收命令。
- 文档更新。

模板见 `COLLAB.md`。

### 4. 验收

验收不是只看测试绿。

必须同时检查：

- 测试是否通过。
- 体感是否符合设计。
- 文案和代码是否一致。
- Wiki / 图标 / 文档是否同步。
- 是否引入新隐式字段。
- 是否有未清理对象。

### 5. 复盘

每个版本结束写 5 行复盘即可：

```md
## vX.Y 复盘
- 做成：
- 没做：
- 新风险：
- 应进规范：
- 下一版建议：
```

长期历史写入 `DEVDOC.md`，新规则写入 `STRUCTURE_RULES.md`。

## 四、质量门槛

### P0 门槛

以下问题必须立刻修：

- 游戏崩溃。
- 页面卡死。
- 选择弹窗无法关闭。
- 起局 / 结算 / 波次推进断裂。
- 长跑固定时间必现错误。
- 移动端无法开始游戏。

### P1 门槛

以下问题应在当前小版本修：

- 遗物或进化越过前置。
- source 误触发。
- Boss 结算误触发。
- 弹体无生命周期。
- UI 遮挡核心信息。
- 每日种子污染普通随机。

### P2 门槛

以下问题可排进下一轮：

- 单个遗物强度偏高。
- 视觉反馈不够明确。
- 图标缺失。
- Wiki 说明不够细。
- 移动端手感微调。

## 五、文档更新机制

每次改动按表同步：

| 改动 | 必更 | 可选 |
|---|---|---|
| 版本完成 | `DEVDOC.md` | `README.md` |
| 阶段方向变化 | `ROADMAP.md` | `PROJECT_INTRO.md` |
| 结构规则变化 | `STRUCTURE_RULES.md` | `ARCHITECTURE.md` |
| 多 agent 协作变化 | `MULTI_AGENT_PROTOCOL.md` | `DOC_INDEX.md` / `COLLAB.md` |
| 当前 agent 任务变化 | `AGENT_BOARD.md` | 无 |
| 日常流程变化 | `DEVELOPMENT.md` | `COLLAB.md` |
| 文档分类变化 | `DOC_INDEX.md` | `README.md` |
| 内容数量变化 | `PROJECT_INTRO.md` | `README.md` / `DEVDOC.md` |
| 美术管线变化 | `ART_DIRECTION.md` / `ASSET_MANIFEST.md` | `ROADMAP.md` |

文档只保留各自职责，不重复粘贴长段：

- 当前事实：`PROJECT_INTRO.md`
- 入口索引：`DOC_INDEX.md`
- 阶段路线：`ROADMAP.md`
- 迭代机制：`ITERATION_SYSTEM.md`
- 多 agent 协作：`MULTI_AGENT_PROTOCOL.md`
- 当前 agent 任务：`AGENT_BOARD.md`
- 结构红线：`STRUCTURE_RULES.md`
- 日常流程：`DEVELOPMENT.md`
- 技术历史：`DEVDOC.md`
- 任务模板：`COLLAB.md`

## 六、准入与冻结

### 可以加内容的条件

同时满足：

- 没有 P0 bug。
- 当前版本测试通过。
- 文档入口未冲突。
- 新内容能写入数据表或通用机制。
- 有测试预算。

### 必须冻结内容的条件

出现任一项：

- 选择逻辑再次出错。
- 49 秒 / 长跑崩溃复发。
- 后期明显卡顿且对象来源不明。
- 移动端基础交互不可用。
- 文档数量和代码数量冲突。
- Claude / agent 按旧任务包执行。

冻结期间只允许修 bug、补测试、写规范和做小型结构收口。

## 七、给接手模型的固定开场

每次把项目交给 Claude / Codex / agent 前，复制这段：

```md
先读：
1. PROJECT_INTRO.md
2. DOC_INDEX.md
3. ROADMAP.md
4. STRUCTURE_RULES.md
5. DEVELOPMENT.md
6. DEVDOC.md 末尾
7. COLLAB.md

当前目标：<一句话>
任务类型：<Bug / 结构 / UI / 性能 / 平衡 / 内容 / 文档>

限制：
- 不按 TASKS.md / TASKS_V9.md / TASKS_V12.md / V5_PREP.md 派活，它们是历史归档。
- 不直接 push 到 g.attacks / g.fires / g.eProj。
- 伤害和击杀必须带 source。
- 选择逻辑必须过滤前置和后续项。
- 新机制必须补测试。
- 本轮不做无关内容扩展。

验收：
- 常规代码改动跑 node --check game.js、node smoke_test.js、node content_test.js。
- 结构或测试改动跑 npm run test:all。
- 文档-only 改动说明未改运行代码。
```

## 八、每周或每阶段例行审计

建议每个阶段至少做一次：

1. 文档入口审计：`DOC_INDEX.md` 是否仍准确。
2. 数量审计：`PROJECT_INTRO.md` 与 `npm run ctx` 是否一致。
3. 规则审计：复发 bug 是否进入 `STRUCTURE_RULES.md`。
4. 测试审计：是否有旧 bug 没有回归测试。
5. 性能审计：长跑对象数是否增长。
6. 图标审计：缺图清单是否更新。
7. 构筑审计：冷标签、死权重、过强组合是否有记录。
