# 墨祟：走阴录 多 Agent 协作协议 v13.1

> 目标：允许 OpenCode / Trae / cc / Claude / Codex 等多个工具参与，但不把维护成本变成新的负担。工具名不绑定固定角色；每轮只声明“本轮身份”。

## 一、核心原则

### 1. 身份按任务定，不按工具定

同一个工具在不同轮次可以承担不同身份：

| 身份 | 适合任务 | 权限 |
|---|---|---|
| 主控 | 拆任务、定优先级、合并、最终验收 | 可决定路线和合并顺序 |
| 实现 | 修 bug、写代码、补测试 | 只改允许文件 |
| 审查 | code review、风险识别、验收建议 | 默认不改代码 |
| UI / 美术 | CSS、图标、HUD、视觉表现 | 不改战斗数值 |
| 内容 | 遗物、敌人、誓印、文案、Wiki | 不碰核心循环 |
| 测试 | 复现、写回归、跑测试 | 不顺手改玩法 |
| 文档 | 路线、规范、交接、版本记录 | 不改运行代码 |

关键不是“哪个工具是什么”，而是“本轮允许做什么”。

### 2. 文件锁比角色更重要

每个任务必须声明：

- 允许修改。
- 禁止修改。
- 锁定文件或锁定函数。
- 必跑测试。

同一时间不建议两个 agent 改同一个核心文件，尤其是 `game.js`。

### 3. 记录要轻，规则要稳

维护文件分三层：

| 层级 | 文件 | 维护粒度 |
|---|---|---|
| 入口 | `DOC_INDEX.md` / `README.md` / `PROJECT_INTRO.md` | 少改，只指路 |
| 稳定规则 | `MULTI_AGENT_PROTOCOL.md` / `STRUCTURE_RULES.md` / `ITERATION_SYSTEM.md` | 规则变化才改 |
| 当前状态 | `AGENT_BOARD.md` | 高频改，只保留当前任务 |

不要把每次聊天、每个想法、每个小失败都写进文档。只有可复发、可执行、会影响后续协作的内容才进入规范。

## 二、协作模式

### A. 单主控模式

适合大多数情况。

```text
主控拆任务 → agent 分头做 → 主控串行合并 → 主控跑测试 → 主控写版本记录
```

规则：

- 只有主控能合并。
- 实现 agent 不能自己扩大任务范围。
- 审查 agent 不直接改代码。
- 任务完成后主控决定是否进入下一步。

### B. 双主控模式

只适合复杂阶段，比如一个管代码，一个管美术/文档。

```text
代码主控：game.js / gamedata.js / tests
表现主控：game.css / assets / docs / wiki
```

规则：

- 两个主控必须锁不同文件域。
- 任一方要改对方文件，先在 `AGENT_BOARD.md` 登记。
- 最终版本收口只能有一个最终验收人。

### C. 审查并行模式

适合让多个模型同时“看看有没有问题”。

```text
实现完成 → 多个审查 agent 并行 review → 主控归并发现 → 决定修哪些
```

规则：

- 审查输出只要 findings，不要重写方案。
- Findings 必须有文件/函数/原因/风险。
- 重复 findings 合并，不按模型数量计权。

## 三、任务板

当前任务只记录在 `AGENT_BOARD.md`。它不是历史档案，只是施工现场白板。

必须记录：

- 任务 id。
- 本轮身份。
- owner。
- 状态。
- 允许文件。
- 锁定区域。
- 验收。

不记录：

- 长篇聊天。
- 已完成很久的任务。
- 大段代码。
- 模型表现评价。

## 四、文件锁规则

### 高风险锁

以下区域同一时间只允许一个实现者改：

| 文件 / 区域 | 原因 |
|---|---|
| `game.js` | 主循环、渲染、状态机集中 |
| `hitE / damageEnemy / onEnemyKilled / hurtP` | source、伤害、击杀会互相影响 |
| `startWave / cleanupWave / spawnWave` | 波次生命周期容易引入隐性 bug |
| 遗物选择 / 进化选择 / 誓印选择 | 前置和后续项容易串 |
| `mobile-controls.js` | 双端输入容易互相污染 |

### 中风险锁

可以并行，但要说明边界：

| 文件 / 区域 | 可并行条件 |
|---|---|
| `gamedata.js` | 不改同一数据表或同一 id |
| `game.css` | 不改同一弹窗 / HUD 区域 |
| `content_test.js` | 测试编号或 block 由主控分配 |
| `wiki.html` | 不和数据结构改动并行 |

### 低风险锁

通常可并行：

| 文件 / 区域 | 说明 |
|---|---|
| 文档 | 只要不同时改同一入口文档 |
| 美术 prompt | 不影响运行 |
| 审计报告 | 不影响运行 |

## 五、任务粒度

推荐粒度：

- 修一个明确 bug。
- 补一个回归测试。
- 优化一个 UI 区域。
- 审计一个系统。
- 生成一批 20-30 个图标清单。
- 整理一份文档入口。

不推荐粒度：

- “优化游戏”。
- “修所有 bug”。
- “加强所有 UI”。
- “补完所有图标”。
- “重构 game.js”。
- “顺手把内容也加了”。

## 六、交付格式

每个 agent 交付必须短，方便主控合并。

```md
## Agent 交付

任务：
身份：
owner：

-

- 

自测：
- 

风险：
-

```

```md
## Review Findings

### Finding 1 [P0/P1/P2]
位置：

```

## 七、主控合并规则

主控每次合并前检查：

1. 是否登记在 `AGENT_BOARD.md`。
2. 是否越过允许文件。
3. 是否触碰锁定区域。
4. 是否补测试。
5. 是否跑了要求命令。
6. 是否需要更新文档。
7. 是否有未解决风险。

合并顺序：

1. P0 bug。
2. 回归测试。
3. 结构收口。
4. UI / 性能。
5. 内容。
6. 文档补充。

如果两个任务冲突，优先保留“测试 + 小改动 + 明确 source”的版本。

## 八、什么时候该写入规范

满足以下任一条件才写进长期规范：

- 同类 bug 复发 2 次以上。
- 多个 agent 都容易犯。
- 会导致崩溃、卡页面、状态错乱。
- 会影响未来任务分配。
- 会影响测试或合并流程。

写入位置：

| 内容 | 文件 |
|---|---|
| 多 agent 协作规则 | `MULTI_AGENT_PROTOCOL.md` |
| 当前 agent 任务状态 | `AGENT_BOARD.md` |
| 代码 / UI / 性能红线 | `STRUCTURE_RULES.md` |
| 版本节奏 / 冻结准入 | `ITERATION_SYSTEM.md` |
| 日常开发命令 | `DEVELOPMENT.md` |
| 版本历史 | `DEVDOC.md` |

## 九、最小开工模板

给任意 agent 派活时，用这个短模板即可：

```md
先读 PROJECT_INTRO.md、DOC_INDEX.md、ROADMAP.md、ITERATION_SYSTEM.md、STRUCTURE_RULES.md、MULTI_AGENT_PROTOCOL.md。

任务 id：
本轮身份：
目标：

允许修改：
- 

禁止修改：
- 

锁定区域：
- 

验收：
- 

交付按 MULTI_AGENT_PROTOCOL.md 的 Agent 交付格式。
```

## 十、维护成本控制

- `AGENT_BOARD.md` 只保留 current active / pending / recently done，不做长历史。
- 完成后的重要经验进 `DEVDOC.md` 或 `STRUCTURE_RULES.md`。
- 旧任务超过一个阶段就从 board 删除。
- 不为每个 agent 建单独长期文档。
- 不记录模型人格评价，只记录可执行事实。
