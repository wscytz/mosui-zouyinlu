# 墨祟：走阴录 路线书 v13.1

> 这份文档决定阶段方向。具体版本节奏、任务流、冻结/准入和复盘机制见 `ITERATION_SYSTEM.md`。

## 当前基线（v14.0）

- **内容体量**：5 武器 / 216 遗物 / 43 进化 / **51 敌人** / 5 Boss / 31 誓印 / 53 成就 / 10 关卡调制器
- **代码规模**：`game.js` 约 5323 行 + `game-utils.js` 16 行 + `gamedata.js` 约 1170 行（RELIC_RULES 已迁入），上限 8000 行
- **技术栈**：Canvas 2D，零框架，`game.js` 单 IIFE，`gamedata.js` 共享数据，`game-utils.js` 纯工具
- **模式**：普通 / 无尽 / 每日种子
- **测试基线**：smoke 44 + content 241 + wave 16 + stress 11 + robust 5 + seeded 3 = 320 项
- **最新重点**：v14.0 软拆分阶段 2 完成 + 阶段 3 加第 5 Boss「墨魇」（数据驱动）

## 总策略

### 先治理，后扩展

v13.0 已经具备较大内容体量。后续如果继续直接加遗物、敌人、Boss，会把 bug 排查、构筑平衡、图标缺口和移动端适配一起放大。

下一阶段优先级：

1. **结构治理**：把运行时生命周期、选择逻辑、对象池、source 判定、文档口径统一。
2. **测试补强**：把过去反复出现的 bug 写成回归测试。
3. **UI / 图标管线**：先建立规则和清单，再批量补图。
4. **移动端发布准备**：先真机验证，再 APK。
5. **内容扩展**：等结构稳定后再开第 6 武器、第 5 Boss 或新大系统。

### 当前禁止事项

- 不新增第 6 武器。
- 不新增第 5 Boss。
- 不批量新增遗物、敌人、誓印、成就。
- 不大改 `hitE / hurtP / spawnWave` 主结构。
- 不把新机制写成大量散落的 `id` 特判。
- 不在没有测试的情况下接入新运行时机制。
- 不为图标一次性生成全部缺口，先建清单和优先级。

## v13.1：结构治理版

**目标**：让现有内容更稳，让后续任务可以被清晰拆分、验收和回滚。

### A. 文档口径统一

| 优先级 | 事项 | 交付物 |
|---|---|---|
| P0 | 同步项目数量 | `PROJECT_INTRO.md`、`ROADMAP.md`、`DEVDOC.md`、`DEVELOPMENT.md` 数量一致 |
| P0 | 更新架构约束 | `ARCHITECTURE.md` 写清 v13 后的模块边界和新增不变量 |
| P1 | 明确任务模板 | `COLLAB.md` 改成当前 Claude / Codex / agent 实际分工 |
| P1 | 整理 bug 追踪 | `DEVDOC.md` 只保留当前未修 bug，已修条目进入版本历史 |

### B. 运行时结构收口

| 优先级 | 事项 | 说明 |
|---|---|---|
| P0 | 对象入池统一 | 攻击、火场、敌弹、粒子都必须走统一入口，禁止直接 `push` |
| P0 | 命中 source 统一 | 近战击杀、魂伤、反伤、召唤物、弹体都必须带来源，避免遗物误触发 |
| P0 | 选择逻辑收口 | 遗物 / 进化 / 誓印的前置、后续项、跳过逻辑必须有统一过滤顺序 |
| P1 | 波次生命周期收口 | `startWave / cleanupWave / waveClear` 明确哪些对象清、哪些对象保留 |
| P1 | Boss 生命周期收口 | Boss 阶段、召唤物、分身、结算触发只走单一入口 |
| P1 | 状态字段登记 | 玩家运行时字段、敌人临时字段、全局计时器建立清单，避免隐式字段扩散 |

### C. 回归测试补强

把历史高发 bug 转成固定测试，不靠人工记忆。

| 类型 | 必测场景 |
|---|---|
| 遗物选择 | 不能给未解锁后续项；跳过后不残留；同波不重复弹出 |
| 进化选择 | 未选前置进化时不能出现后续进化 |
| 弹体命中 | 穿透弹同一目标不重复逐帧结算；折返弹必须初始化命中表 |
| source 判定 | 近战遗物只吃近战击杀，魂/火/反伤不能误触发 |
| 诱饵逻辑 | 敌人移动目标和攻击目标一致 |
| 每日种子 | 同日期同选择可复现，不污染普通模式 RNG |
| 长跑稳定 | 49 秒、3 分钟、无尽 10 波无 console error |

### D. 验收口径

v13.1 完成必须满足：

```bash
node --check gamedata.js
node --check game.js
node --check mobile-controls.js
node smoke_test.js
node content_test.js
npm run www
```

如果测试脚本已经扩展，按 `npm run test:all` 为准。

文档-only 改动可以不跑全量测试，但交付说明必须写明“未改运行代码”。

**目标**：把画风、图标、HUD、弹窗体验整理成可批量生产的系统，而不是逐个补丁。

## v13.3：性能与移动端发布版

**目标**：把“后期卡卡的”变成可观测、可定位、可限制的问题。

### A. 性能预算

| 对象 | 规则 |
|---|---|
| 攻击对象 | 必须有生命周期、命中表、速度异常清理 |
| 粒子 | 必须有数量上限和低性能档降级 |
| 火场 / 墨池 | 必须有上限、合并或过期策略 |
| 敌弹 | 必须有离屏清理和最大存活帧 |
| 浮字 | 必须有上限，不能在高频伤害中无限生成 |

### B. 观测工具

| 优先级 | 事项 | 说明 |
|---|---|---|
| P0 | debug HUD | 显示 FPS、敌人、攻击、粒子、火场、敌弹数量 |
| P0 | 长跑测试 | 固定种子跑 3 分钟，记录对象峰值 |
| P1 | 压力场景 | 铃、伞、穿透、弹道折返、墨夜、无尽组合压测 |
| P1 | 移动性能档 | 根据设备或手动选项降低粒子和墨雾 |

### C. APK 前置

APK 打包前必须完成：

- 移动端横屏可玩。
- 右摇杆瞄准不会误触。
- 弹窗不会卡住战斗状态。
- 低性能档存在。
- 结算、排行榜、本地存档可用。
- `npm run www` 和 Capacitor 同步流程文档化。

## v14.0：模块化与大内容前置

**目标**：为第 6 武器、第 5 Boss、更复杂关卡机制释放维护空间。

### A. 软拆分优先

不要一上来强拆 `game.js`。先做低风险软拆分：

| 阶段 | 事项 | 说明 |
|---|---|---|
| 1 | 常量和调参集中 | `CAPS / LIMITS / RANGES / TUNING` 统一位置 |
| 2 | 纯函数外移 | 分数、格式化、选择过滤、权重计算优先外移 |
| 3 | UI 辅助外移 | DOM 弹窗、toast、截图文案逐步脱离核心循环 |
| 4 | 调试工具外移 | debug HUD、性能统计、测试钩子不塞进战斗逻辑 |
| 5 | 再考虑运行时拆分 | 只有边界稳定后才拆 `game-core / game-render / game-ui` |

### B. 模块边界建议

未来可以演进为：

```text
gamedata.js          内容数据
game.js              启动、主循环、状态机
game-combat.js       攻击、命中、伤害、击杀
game-wave.js         波次、敌人生成、Boss 生命周期
game-render.js       Canvas 绘制
game-ui.js           DOM 弹窗、HUD、结算
game-debug.js        debug HUD、性能统计
mobile-controls.js   移动端输入
```

拆分原则：

- 每次只拆一个边界。
- 拆前加测试，拆后行为不变。
- 不在拆分同时新增玩法。
- 保持脚本加载顺序明确。
- 不引入框架，不改成构建依赖项目。

### C. 大内容解锁条件

第 6 武器 / 第 5 Boss / 新大系统必须满足：

- v13.1 结构治理完成。
- v13.2 UI / 图标管线完成第一批。
- v13.3 性能观测和移动端基础验证完成。
- `game.js` 行数仍有安全余量，或已完成至少一轮软拆分。
- 新内容能写进数据表和通用机制，不靠大段特殊分支。

## Claude / agent 执行规则

详细结构红线、代码规范、UI/图标规范、测试规范见 `STRUCTURE_RULES.md`。`ROADMAP.md` 只决定阶段节奏，具体执行以规范文档为准。

### 任务分发模板

```md
目标：<版本号 + 一句话目标>

范围：
- 允许修改：
- 禁止修改：

必须遵守：
- 保持 IIFE / var / function / 双引号风格。
- 不直接 push 到 g.attacks / g.fires / g.eProj。
- 敌人扣血统一走 damageEnemy(g,e,dmg,src)。
- 新机制必须带 source 判定。
- 新遗物必须同步 PREREQS / BUILD_PREFS / RELIC_RULES / 图标 / 测试。
- 不要大改 hitE / hurtP / spawnWave。

验收：
- node --check gamedata.js
- node --check game.js
- node smoke_test.js
- node content_test.js
- npm run www

交付：
- 变更清单
- 自测结果
- 未解决问题
- 文档更新位置
```

### 审核重点

1. 有没有新增隐式字段但没默认值。
2. 有没有直接 `push` 到运行时对象池。
3. 有没有 source 判定不清导致遗物误触发。
4. 有没有选择池前置过滤漏掉后续项。
5. 有没有 UI 弹窗改变状态机。
6. 有没有热路径 `Math.sqrt()`、无限粒子、无限浮字。
7. 有没有只更新文案不更新 wiki / 测试 / 图标。

## 红线

- 修 bug 优先级永远高于加内容。
- 每个版本只解决一个主问题。
- 内容数量增长不能掩盖结构问题。
- 没有测试的机制视为未完成。
- 文档数量、wiki 展示、测试数量必须和实际代码同步。
- Claude / agent 不能凭记忆改项目，必须先读当前文档和脚本输出。
